## UI-01 [P1] エンジニア詳細画面のJavaScriptが構文エラーで全停止する

- 対象画面: `/engineer/detail`
- 対象ファイル／関数: `static/js/modules/engineer-detail.js` の `deleteCareer(id)`、464行目
- 検出方法: 同梱Node.jsによる `node --check`
- 発生条件: エンジニア詳細画面で同JSを読み込む。
- 実際の結果: `SyntaxError: missing ) after argument list`。`if` ブロックの閉じ括弧が1つ多く、ファイル全体がパースされない。詳細表示、写真、スキル、経歴、営業担当など同ファイル内の全機能が実行されない。
- 期待結果: 余分な `}` を削除し、全JSファイルの構文検査が成功する。画面初期化と各操作が実行可能になる。
- 追加すべきテスト: CIで26ファイルすべてへ `node --check` を実行し、さらに詳細画面のスモークテストで初期データ読込を確認する。

## UI-05 [P1] 案件スキル保存失敗でもモーダルを閉じ、保存完了のように見せる

- 対象画面: `/project/list` の案件登録／編集モーダル
- 対象ファイル／関数: `project.js` の `saveProject()`、`saveProjectSkills()`、`finishSave()`
- 発生条件: 案件本体保存後、スキル置換APIが失敗する。
- 実際の結果: エラートースト表示後にも `finishSave()` を呼び、モーダルを閉じ、入力を消去して一覧を再読込する。案件本体だけ保存された部分成功をユーザーが見落としやすく、再試行情報も失われる。
- 期待結果: 単一API・単一トランザクションにする。分離を維持する場合はモーダルを開いたまま入力を保持し、「案件本体は保存済み、スキルは未保存」を明示して再試行できるようにする。
- 追加すべきテスト: スキルAPIを500にして、モーダル・入力保持・部分成功メッセージを検証する。

## UI-07 [P2] UTCの `toISOString()` によりJST早朝の日付・月が前日にずれる

- 対象画面／ファイル:
  - 稼働予定: `availability-calendar.js` 14、15、88行付近
  - 候補者: `candidate.js` 104行
  - 顧客活動: `customer-detail.js` 86行、`templates/customer/detail.html` 151、155行
  - 請求: `invoice.js` 50、85、217行付近
- 発生条件: 日本時間00:00～08:59、特に月初に画面を開く／日付を初期設定する。
- 実際の結果: `new Date().toISOString()` はUTC日付を返すため、JSTの当日が前日になり、月初は前月になる。期限超過判定、活動日、入金日、対象月の初期値がサーバーの `LocalDate.now()` とずれる。
- 期待結果: `getFullYear/getMonth/getDate` を使うローカル日付ヘルパーを `SES.date` 等へ共通化し、全画面で使用する。
- 追加すべきテスト: タイムゾーンをAsia/Tokyoに固定し、月初00:30、通常日08:59、09:00で期待日付を検証する。

## UI-08 [P1] 円で保存・描画する金額に「万円」ラベルを付けている

- 対象画面:
  - ダッシュボード売上・粗利チャート
  - 案件登録モーダルの単価入力
  - AI画面のエンジニア単価選択
- 対象ファイル:
  - `messages.properties` の `dashboard.chart.sales_label/profit_label`、`project.modal.price_min/max`
  - `dashboard.js` の `renderCharts(...)`
  - `project.js` の保存値
  - `ai.js` 160行付近
- 発生条件: 例として800,000円の単価・売上を表示する。
- 実際の結果: バックエンド集計と保存値は円で、チャートも生の円を描画するがラベルは「万円」。案件入力例も600000／800000なのに「万円」。`ai.js` は800000へ直接「万円」を付ける。最大1万倍の誤読になる。
- 期待結果: 内部単位を円に統一するなら全ラベルを円にし、表示時に桁区切りする。万円表示を採用する箇所は値を `/10000` し、入力時は `*10000` する。どちらかを画面横断で統一する。
- 追加すべきテスト: 800,000円を全対象画面で表示し、期待表示と保存後DB値が一致することを検証する。

## UI横断の受入基準

1. 全JSファイルの `node --check` をCI必須にする。
2. 主要画面はja/en/zh-CN/ko、Asia/Tokyoの日付境界、データ0件／11件超／大量件数をテストする。
3. 保存操作は全成功・全失敗・部分失敗を明確に区別し、失敗時に入力を失わない。
4. 金額の内部単位と表示単位を画面ごとではなくシステム全体で統一する。
5. `.html()` へ動的値を渡す場合は明示的なエスケープを必須とし、可能な限りDOM生成と `.text()` を使う。
6. バックエンドに存在する主要ユースケースは、対応画面から権限内で完結できることをE2Eで確認する。
