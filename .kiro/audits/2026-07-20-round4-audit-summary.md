## 1. 結論

モジュール間で13件（P1:4 / P2:6 / P3:3）、モバイルUIで18件（P1:2 / P2:7 / P3:9）、**計31件**を記録した。

特に重大なのは「**ページ丸ごと動いていない**」系の4件で、いずれも本監査でアプリを実際に起動・ログインして確認済み:

1. **X-01**: `/sales-performance`（営業成績）— `layout:fragment="scripts"` が base.html の `page-js` スロットと不一致で**JSが一切読み込まれない**＋未ロードの axios に依存。画面は静的な「データがありません」のまま。
2. **X-02**: `/ai/matching`（AIアシスタント）— 同じフラグメント名不一致で ai.js が読み込まれず、**チャットが「入力中...」アニメーションのまま永久停止**。ボタンに生の `th:text=...` テキストも表示される。
3. **X-03**: `/analytics/availability-calendar`（稼働率カレンダー）— base.html に存在しないフラグメント署名を参照しており **HTTP 500**。さらにどのメニュー・画面からもリンクされていない孤児ページ。
4. **X-04**: Flyway validate 有効化後、**編集済みV37を適用済みの既存ローカルDBでアプリが起動不能**（本監査の起動時に実際に発生。`flyway repair` で回復することを確認）。

モバイル側の最重要は **M-01**（カンバンがタッチ端末でスクロール不能、スワイプが誤ドラッグになり**ステータス変更APIが誤発火**する）と **M-02**（エンジニア本人のスマホ利用が前提のマイ勤怠で、日次入力フォームが1行6列に圧縮され入力不能）。

モジュール間では、画面間ナビゲーション契約の破れ（X-05 月次締め→勤怠/請求のパラメータ無視、X-06 ダッシュボード→要員一覧の status 無視）と、**単価改定が未確定勤怠の金額を再計算しないまま確定・請求へ流れる X-07** が実データの金額に影響し得る。

## 3. 検証記録

### 3.1 実機検証（今回実施）

ローカルMySQL上でアプリを起動し、admin ログイン済みセッション（curl）で確認した:

- `GET /sales-performance` → 200 だが描画HTMLに `sales-performance.js` と i18n 定数が**不存在**（X-01確定）
- `GET /ai/matching` → 200 だが `modules/ai.js` が不存在、本文に生の `th:text="#{common.btn.saveSettings}"` 文字列が出力（X-02確定）
- `GET /analytics/availability-calendar` → **HTTP 500**（X-03確定）
- 初回起動時に `FlywayValidateException: Migration checksum mismatch for migration version 37` で起動失敗（X-04確定）

**環境への操作の記録**: 検証のため CLAUDE.md 記載の手順どおり `flyway-maven-plugin:repair` をローカルDB（ses_manager_db）へ実行した（`flyway_schema_history` のchecksum再整列のみ。データ・スキーマは不変更）。検証後、起動したアプリのプロセスは停止済み。ソースコードへの変更は本監査ドキュメント3ファイルの追加のみ。

### 3.2 静的照合

- 全26 JSモジュール×全34テンプレートを、レイアウト契約（`layout:fragment` 名）・画面間リンクのクエリパラメータ授受・API既定値・i18nキーの観点で突合。
- JSが参照する静的i18nキー約300件を4ロケールの定義キーと機械突合（未定義は `invoice.btn.paymentHistory` の1件のみ = M-07）。
- 更新系 fetch の CSRFヘッダー付与は全モジュールで漏れなしを確認。
- ブレークポイント整合（common.css 992px ⇔ Bootstrap `d-lg-none`）は机上計算で境界不一致を確定（M-06）。

### 3.3 未実施（次回への持ち越し）

## 5. 横断的な再発防止策（強く推奨）

1. **ページ共通レンダリングテスト**: 全 `*PageController` ルートをMockMvcで描画し、(a) HTTP 200、(b) 対応する `modules/*.js` の script タグが含まれる、(c) 本文に `th:text=` という文字列が漏れていない、の3点をアサートする。X-01/X-02/X-03 と M-10 はこの1本で全て検出できた。
2. **`layout:fragment` 名の規約明文化**: 使用可能なスロットは `content` / `page-css` / `page-js` の3つのみであることを CLAUDE.md のフロントエンド節へ追記する（`scripts` という誤記が2ファイルで独立に発生している）。
3. **適用済みマイグレーションの編集禁止**を CLAUDE.md に明記する（X-04 の根本原因）。
4. **画面間リンクの契約**: クエリパラメータを渡すリンクを追加する際は、受け側JSの `URLSearchParams` 処理をセットで実装・テストする規約とする（X-05/X-06/M-18、および X-02 の engineerId が同型）。
