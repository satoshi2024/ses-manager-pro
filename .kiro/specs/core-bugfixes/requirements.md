# Requirements: ログイン・CRUD操作の不具合修正

## 背景

2026-07-11 の調査で判明した事実:

- ローカル環境の MySQL(3306) が起動しておらず、8080 で稼働中のアプリは DB 接続に失敗する状態だった。この状態で admin/admin123 でログインを試みると「ユーザーIDまたはパスワードが正しくありません」と表示される（実際は DB 未接続によるエラーであり、パスワード誤りではない）。
- `sql/002_init_master_data.sql` に初期管理者 `admin / admin123` が平文パスワードで登録されている（`SecurityConfig.passwordEncoder()` が `NoOpPasswordEncoder` のため平文比較）。
- 各画面（要員/顧客/案件/契約/提案/メールテンプレート）の新規登録・編集・削除は `bootstrap.Modal.getInstance(...).hide()` と `Toast` 表示を実装済みだが、いずれも `res.code === 200` の分岐内でのみ実行される。バックエンドが 500 やセッション切れの HTML を返すと弾窗が閉じず、ユーザーには何も起きていないように見える。
- `common.js` のグローバル Ajax ハンドラ（288〜300行）が「200 かつ Content-Type が text/html」を一律セッション切れ扱いにしており、`/api/` を含まない通常ページの通信でも誤検知しうる条件になっている。
- 通知ドロップダウンの各項目リンクは `href="#"` で遷移先が無い。`window.matchAI` はダミー実装のまま。

## 要件一覧

### Requirement 1: ログイン失敗理由の切り分け
ログインできない時、ユーザーは「パスワードが違う」のか「システム側の問題（DB未接続等）」なのかを判別できなければならない。

**受入基準:**
1. DB 接続不可などのシステム例外発生時、ログイン画面には認証失敗と異なる文言（例:「システムに接続できません。時間をおいて再試行してください」）を表示すること。
2. 通常の ID/パスワード誤りの場合は従来通り「ユーザーIDまたはパスワードが正しくありません」を表示すること。
3. サーバーログに例外の原因（DB接続失敗 vs 認証失敗）が区別して記録されること。

### Requirement 2: ログイン成功時のフィードバック
ログイン成功後、ユーザーはログインが成功したことを画面上で確認できなければならない。

**受入基準:**
1. `/dashboard` へリダイレクト後、初回表示時に「ログインしました」の成功トーストが1回表示されること。
2. ページリロードや別画面遷移で再度表示されないこと。

### Requirement 3: CRUD操作（登録・編集・削除）の弾窗と通知の一貫性
要員/顧客/案件/契約/提案/メールテンプレートいずれの画面でも、保存・削除操作の結果（成功/失敗）が必ず画面に反映されなければならない。

**受入基準:**
1. 保存/削除が成功した場合: 対象モーダルが自動的に閉じ、成功トーストが表示され、一覧が再読み込みされること。
2. 保存/削除が失敗した場合（バリデーションエラー、外部キー制約、500エラー、セッション切れ含む）: モーダルは閉じずに残り、具体的なエラーメッセージがトーストで表示されること。
3. バックエンドが例外を投げた場合でも常に `ApiResult` 形式の JSON（HTML ではない）を返すこと。
4. 6画面（engineer, customer, project, contract, proposal, email-template）すべてで同一の挙動になること（個別実装のばらつきをなくす）。

### Requirement 4: セッション切れ誤検知の修正
セッション切れの検知条件が実際にセッション切れでないケースを誤検知しないこと。

**受入基準:**
1. `/api/` 配下以外の通常ページ遷移で誤って「セッションが切れました」トーストが出ないこと。
2. 実際にセッションが切れて `/login` にリダイレクトされた場合は正しく検知され、トースト表示後に `/login` へ遷移すること。

### Requirement 5: パスワードのハッシュ化
パスワードは平文で保存・比較しないこと。

**受入基準:**
1. `PasswordEncoder` が `BCryptPasswordEncoder` に変更されること。
2. 初期データ（admin ユーザー）のパスワードが BCrypt ハッシュに置き換わること。
3. 変更後も admin / admin123 でログインできること（ハッシュ化後の値で比較される）。

### Requirement 6: 開発環境の再現性
DB 未起動によって「バグ」に見える現象を今後防ぐため、開発環境の起動手順を明文化する。

**受入基準:**
1. README もしくは docs に MySQL 起動手順（または docker-compose）が記載されること。
2. `target/` ディレクトリが `.gitignore` に追加され、以後の差分に含まれないこと。

## 対象外（別途対応・本specでは扱わない）

- 通知ドロップダウンの遷移先実装、AIマッチング(`matchAI`)の本実装、甘特图/収益ダッシュボードの機能拡充 — 別specで扱う。
