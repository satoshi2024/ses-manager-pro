# Requirements Document

## Introduction

現在のSES Manager Proは `admin` 1アカウントのみで運用されており、アプリ上でユーザーアカウントを増減・編集する手段が存在しない。本要件は、管理者がアプリのUIから複数のユーザーアカウントを管理し、ロールごとにアクセス可能な画面（メニュー）を設定できるようにするためのものである。対象は以下の3点。

1. ユーザーアカウントのCRUD（登録・一覧・編集・有効/無効化・削除）とロール割当
2. ロール別メニューアクセス権限の設定（管理者がロールごとに閲覧可能な画面を選択できる）
3. ユーザー管理・権限設定画面自体への管理者専用アクセス制御

パスワードは開発環境では既存仕様通り平文比較（`NoOpPasswordEncoder`）を維持し、本番環境（`prod`プロファイル）ではBCryptハッシュ化に切り替える。

## Requirements

### Requirement 1: ユーザーアカウントCRUD

**目的:** 管理者として、システムを使う従業員ごとにログインアカウントを発行・管理したい。

#### Acceptance Criteria
1. 管理者が `/user/list` にアクセスすると、`sys_user` の一覧（ログインID、氏名、ロール、メールアドレス、ステータス）がページネーション付きで表示される。
2. 管理者が新規登録フォームでログインID・パスワード・氏名・ロール・メールアドレスを入力し保存すると、パスワードは `PasswordEncoder` でエンコードされた上で新規レコードとして保存される。
3. 管理者が既存ユーザーを編集する際、パスワード欄を空のまま保存すると既存パスワードが維持される。
4. 管理者がユーザーのステータスを無効化に切り替えると、対象ユーザーは以降ログインできなくなる（`CustomUserDetailsService` の enabled 判定に従う）。
5. 管理者がユーザーを削除すると、論理削除（`deleted_flag=1`）される。
6. 管理者が自分自身のアカウントを削除または無効化しようとした場合、操作は拒否されエラーメッセージが返る。

### Requirement 2: ロール別メニューアクセス権限

**目的:** 管理者として、ロール（管理者/営業/HR/マネージャー）ごとに閲覧可能な画面を制御したい。

#### Acceptance Criteria
1. ユーザー管理画面の「ロール別メニュー権限」タブでロールを選択すると、全メニュー項目がチェックボックスで表示され、そのロールに許可されているメニューにチェックが入る。
2. 管理者がチェック状態を変更して保存すると、`t_role_menu` の当該ロールの行が置き換えられる。
3. ログインユーザーのサイドバーには、そのユーザーのロールに許可されたメニューのみが表示される。
4. 許可されていないメニューのURLに直接アクセスした場合、画面は403相当のレスポンスとなり表示されない（サイドバー非表示だけに依存しない）。

### Requirement 3: ユーザー管理画面への管理者専用アクセス制御

**目的:** 管理者以外のロールのユーザーが、アカウント管理・権限設定を行えないようにしたい。

#### Acceptance Criteria
1. 管理者以外のロールでログインしたユーザーが `/user/**`, `/api/users/**`, `/api/role-menus/**` にアクセスした場合、アクセスが拒否される。
2. 管理者ロールのユーザーのみ、サイドバーに「ユーザー管理」メニューが表示される（Requirement 2 のメニュー権限機構に基づく）。

### Requirement 4: パスワード保存方式の環境別切替

**目的:** 開発時の利便性を保ちながら、本番環境ではパスワードを安全に保存したい。

#### Acceptance Criteria
1. `prod` プロファイルが非アクティブな環境（開発・テスト）では、パスワードは平文のまま比較・保存される（既存の `admin/admin123` 運用と互換）。
2. `prod` プロファイルがアクティブな環境では、パスワードは BCrypt でハッシュ化されて保存・比較される。
3. アプリケーションコードはプロファイルの違いのみで動作を切り替え、ユーザー登録・更新のロジック自体は変更を要さない。
