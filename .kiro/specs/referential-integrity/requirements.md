# Requirements — 参照整合性ガード + 契約番号採番修正(referential-integrity)

論理削除(`deleted_flag`)運用のため DB の `ON DELETE RESTRICT` 外键が機能せず、
参照されているマスタを API 直叩きで削除できてしまう問題のサーバー側修正。
あわせて契約番号採番の count 方式による衝突バグを修正する。

対象コード: `EngineerServiceImpl` / `CustomerServiceImpl` / `ProjectServiceImpl` / `ContractServiceImpl` / `ContractMapper`。
**コントローラーは編集しない** — 既存の `removeById` 呼び出しがそのまま通る位置(service の
`removeById(Serializable)` オーバーライド)でガードする(WS-D が `ContractApiController` を編集するため)。

## R1. 要員の削除ガード

現状: `/api/contracts/check-active` によるチェックはカンバン画面の JS からしか呼ばれず、
`DELETE /api/engineers/{id}` を直接叩けば稼動中契約を持つ要員も削除できる
(契約・実績・請求が「不明」要員として宙に浮く)。

受け入れ基準:
1. `稼動中` の契約が1件でも紐づく要員の削除は `BusinessException`
   「稼動中の契約があるため削除できません」で拒否される。
2. 進行中の提案(ステータスが `成約`/`見送り` 以外)が紐づく要員の削除は
   「進行中の提案があるため削除できません」で拒否される。
3. 上記に該当しない要員は従来どおり論理削除できる(既存テスト
   `EngineerDeleteIntegrationTest` が引き続きグリーン)。

## R2. 顧客の削除ガード

受け入れ基準:
1. 削除されていない案件(`t_project`)が紐づく顧客の削除は拒否される。
2. 削除されていない契約(`t_contract`)が紐づく顧客の削除は拒否される。
3. 取消されていない請求書(`t_invoice.deleted_flag=0`)が紐づく顧客の削除は拒否される。
4. メッセージは何が紐づいているかを明示する(例:「案件が3件紐づいているため削除できません」)。

## R3. 案件の削除ガード

受け入れ基準:
1. 削除されていない契約が紐づく案件の削除は拒否される。
2. 進行中の提案(`成約`/`見送り` 以外)が紐づく案件の削除は拒否される。

## R4. 契約の削除ガード

受け入れ基準:
1. 実績(`t_work_record`)が1件でも存在する契約の削除は
   「実績が登録されているため削除できません」で拒否される(実績・請求・BP支払の親を消させない)。
2. `稼動中` の契約は削除できない(先に終了/解約への状態変更を要求する)。
3. 実績がなく稼動中でもない契約は従来どおり論理削除できる。

## R5. 契約番号採番の衝突修正

現状: `saveWithBusinessRules` が `selectCount+1` で採番するため、契約を論理削除すると
count が減る一方 UNIQUE 索引には旧番号が残り、以後の新規作成が必ず衝突する。
リトライは `count+1+i` の3回のみで、同月内の削除が3件を超えると採番が恒久的に失敗する。

受け入れ基準:
1. 採番は削除済みを含む当月プレフィックスの最大番号+1 になる(`InvoiceServiceImpl` の方式に合わせる)。
2. 同月の契約を複数論理削除した後でも新規契約が一発で作成できる。
3. 既存のリトライ(マルチインスタンス競合対策)は維持される。

## R6. フロントの体験(補足)

受け入れ基準:
1. 拒否時、既存の各画面(engineer.js / customer.js / project.js / contract.js)は
   `ApiResult.message`(`GlobalExceptionHandler` が生成)を Toast 表示する既存挙動で
   そのまま日本語エラーが出る。**フロント側の改修は不要である**ことをテストで担保する
   (`removeById` が false ではなく例外で拒否されるため、`code != 200` 経路に乗る)。
