# 顧客視点の機能提案（customer-feature-proposals）

2026-07-17 時点のシステム（金銭フロー・ライフサイクル整合性の是正完了後）を、
**SES会社の実際の利用者**——営業・管理部（請求/支払担当）・経営層・要員本人——の視点で棚卸しし、
「次に欲しくなる機能」を優先度付きで提案するドキュメント。

**位置づけ**: 本書は spec 化前の提案段階。発注者が採用を決めた項目ごとに、リポジトリ規約どおり
`.kiro/specs/<name>/`（requirements / design / tasks）を新設して着手する。
本書の「実装概要」は規模感の見積もりのためのスケッチであり、design ではない。

## P1. 作業報告書（月報）と要員セルフサービス勤怠 【規模L・最優先候補】

- **誰のどんな困りごと**: 現状の勤怠は管理部が全要員の月次合計時間を代理入力する設計。実運用では
  要員→(Excel/客先勤怠システム)→営業→管理部と勤務実績が転記されており、二重入力・転記ミス・
  月初の回収催促が SES バックオフィスの最大の作業負荷。さらに客先へ提出する「作業報告書（月報）」
  を別途 Excel で作っているのが通例。
- **現状のギャップ**: `t_work_record` は契約×月の合計値のみ。日次データ・提出/承認の概念・
  要員本人のアカウントが存在しない。
- **実装概要**:
  1. 要員ロール（`ROLE_ENGINEER`）と要員↔`sys_user` の紐付けを追加（freee連携の
     `t_freee_employee_link` と同型の対応表）。要員向け画面はメニュー権限の既存機構
     （`m_menu`/`t_role_menu`/`MenuPermissionFilter`）にそのまま乗る。
  2. 新テーブル `t_work_record_daily`（work_record_id, work_date, start/end, 休憩, 備考）。
     月次画面で日次入力→合計を既存 `t_work_record.actual_hours` へ集約（**精算・請求の既存
     ロジックは無変更**——合計値の生成元が変わるだけ、という接続にする）。
  3. 提出→差戻し/承認の軽量ステートを `t_work_record` に追加（入力中→提出済→承認済(=確定)）。
     既存の 確定/解除・請求済みガードはそのまま流用。
  4. 作業報告書PDF（客先提出用）: `InvoicePdfServiceImpl` のPDF基盤を流用し日次明細+合計+承認印欄。
- **既存資産との接続**: 精算(`SettlementCalculator`)・請求・BP支払・集計口径
  (`MonthlyRevenueCalcService`)は合計値を読むだけなので影響なし。通知（提出依頼・差戻し）は
  `NotificationLinks`/`NotificationGenerateService` に追加。
- **リスク・論点**: 要員アカウントの発行運用（BP要員に発行するか）/ 日次と月次合計の整合
  （日次入力時は合計を自動計算にし手入力を禁止するか）を requirements で確定する必要がある。

## P2. 債権管理（入金消込・エイジング・督促） 【規模M】

- **誰のどんな困りごと**: 請求書は発行できるが、入金は `paid_date` を手で立てるだけ。
  一部入金・過入金の扱いがなく、「どの顧客がいくら遅れているか」を見る画面がない。
  月末の資金繰り確認と督促は台帳外（Excel/記憶）で回っている。
- **現状のギャップ**: `t_invoice` に入金額カラムなし。エイジング集計・督促の仕組みなし
  （`INVOICE_OVERDUE` 通知は社内ベル通知のみ）。
- **実装概要**:
  1. 新テーブル `t_invoice_payment`（invoice_id, 入金日, 入金額, 手数料, 備考）——複数入金対応。
     合計が total に達したら 入金済 へ自動遷移（既存の状態機械 `ALLOWED` を拡張し「一部入金」を追加）。
  2. エイジングレポート画面: 顧客×経過区分（期限内/超過1-30/31-60/61-90/90+日）の残高マトリクス
     ＋ドリルダウン。Excel出力は `ExcelExportService` に1シート追加。
  3. 督促メール: 既存 `MailService.sendWithTemplate` ＋ メールテンプレート機能をそのまま使い、
     請求書一覧から対象を選んで送信・送信履歴は既存 `t_mail_delivery` に記録。
- **既存資産との接続**: 請求状態機械・`calcDueDate`・メール基盤・Excel基盤がすべて既存。
  新規はテーブル1本と画面1枚が中心で費用対効果が高い。
- **リスク・論点**: 過入金・相殺の扱い / 入金消込の取消可否（誤登録の訂正）を requirements で確定。

## P3. 月次締めチェックリスト 【規模S〜M】

- **誰のどんな困りごと**: 月次の締めに必要な確認——「全稼動契約に工数が入っているか」
  「確定済みだが未請求の実績はないか」「未払BPは・期限超過請求はいくつか」——が
  勤怠・請求・BPの3画面に分散しており、締め漏れは人の注意力で防いでいる。
- **現状のギャップ**: 横断ビューなし。確定(`confirmMonth`)は月単位で存在するが
  「締めた」という宣言・ロックの概念がない。
- **実装概要**: 対象月を選ぶと以下を1画面に集約する `/monthly-closing`（新メニュー）:
  工数未入力の稼動契約一覧 / 入力中のまま残っている実績 / 確定済み未請求実績（顧客別）/
  未払BP一覧 / 期限超過請求一覧。各行から該当画面へ遷移。すべてゼロになったら
  「締め完了」を記録（`m_system_config` に `closing.confirmed-months` を持つ最小実装で開始し、
  ロック強化——締め済み月の reopen 拒否など——は第2段とする）。
- **既存資産との接続**: 必要なクエリはすべて既存テーブルの組み合わせで、新テーブル不要。
  既存の reopen ガード群（請求済み・支払済BP・手動階層）が既にロックの大半を担っている。

## P4. 見積書発行 【規模S〜M】

- **誰のどんな困りごと**: SESの商流は 見積書→注文書→（契約・注文請書）で始まるが、システムは
  提案（カンバン）と契約の間に商務書類がなく、見積書は Word/Excel で別管理。単価・精算幅の
  転記ミスが契約時に混入する。
- **現状のギャップ**: 見積の概念・テーブル・帳票が一切ない（コード検索で確認済み）。
- **実装概要**: `t_quotation`（提案ID任意紐付け、顧客、要員/案件、単価、精算幅、有効期限、
  採番 `Q-YYYYMM-NNNN` は請求書採番のリトライ方式を流用）＋見積書PDF
  （`InvoicePdfServiceImpl` の会社情報・角印レイアウトを流用）。受注時に「見積から契約ドラフト生成」
  で単価・精算幅を引き継ぎ、`createDraftFromProposal` と同じ合流点に乗せる。
- **リスク・論点**: 注文書・注文請書まで踏み込むか（CloudSign 連携 spec との接続）は
  第2段として切り離すのが無難。

## P5. 売上着地予測（パイプライン加重） 【規模S】

- **誰のどんな困りごと**: ダッシュボードの「見込み」は確定契約ベースのみ。経営層が知りたい
  「今月・来月の着地」には、提案パイプライン（結果待ちの案件が決まったら+いくら）が入っていない。
- **現状のギャップ**: 提案には `proposed_unit_price` があるが売上予測に一切使われていない。
- **実装概要**: ステージ別の受注確率（書類選考中20%・一次面接40%…）を `m_system_config` で
  管理者が設定し、`確定契約ベース（既存の共通口径） + Σ(オープン提案の提示単価×確率)` の
  加重予測をダッシュボードに点線系列で重ねる。新テーブル不要。
  `MonthlyRevenueCalcService` に加算はせず**別系列**として返す（実績口径を汚さない）。
- **リスク・論点**: 開始月の仮定（成約→翌月1日開始という既存ドラフト規約を流用）を明記する。

## P6. 契約単価の改定履歴（期間別単価） 【規模M〜L】

- **誰のどんな困りごと**: 期中の単価改定（4月から+5万円）は現状「契約を編集して上書き」しかなく、
  改定前の月を再計算・再精算すると新単価で計算されてしまう（過去実績は確定済み金額で守られるが、
  reopen した瞬間に新単価が適用される）。改定履歴も残らない。
- **現状のギャップ**: `t_contract.selling_price/cost_price` は現在値のみ。
- **実装概要**: `t_contract_price_history`（contract_id, 適用開始月, selling/cost, 改定理由）を追加し、
  `saveHours` の精算時に「対象月に有効な単価」を解決するリゾルバを `SettlementCalculator` の
  手前に挟む。契約編集で単価を変えたら「いつから適用か」を必須入力に。
- **リスク・論点**: 影響範囲が精算・ダッシュボード・営業成績のフォールバック
  （契約単価を読む箇所すべて）に及ぶため、**「月次金額の解決」を共通サービスに寄せてある現構造
  （`MonthlyRevenueCalcService.resolveContractAmount`）を活かして一点改修にできるか**が設計の肝。
  代替の軽量案: 履歴テーブルは記録のみ（精算は現行どおり現在値）から始める。

## P7. データスコープ権限（担当データのみ閲覧） 【規模M】

- **誰のどんな困りごと**: 現在の権限はメニュー単位で、営業ロールなら全顧客・全契約・全単価が見える。
  営業間で顧客情報・粗利を見せたくない、という組織要求は SES で頻出。
- **現状のギャップ**: `MenuPermissionFilter` は URL 単位のみ。行レベルの絞り込みなし。
- **実装概要**: `m_system_config` のフラグ（`scope.sales-own-data-only`）で有効化するオプトイン方式。
  有効時、営業ロールのリスト系 API（契約・顧客・要員・営業成績）に
  「自分が担当（契約の sales_user_id / 要員の現任担当 / 顧客の担当営業）」条件を共通的に注入する。
  管理者・マネージャーは常に全件。
- **リスク・論点**: 検索・集計・Excel出力・通知など**読み経路の網羅**が必要で漏れると意味がない。
  導入するなら専用 spec で読み経路の棚卸しから始めること（本提案では最下位に置く理由）。

## 推奨順序と理由

1. **P2 → P3 を先行**（合わせて1〜2サイクル）: 新テーブル1本＋画面中心で小さく、
   「金の回収」という経営直結の穴を塞ぐ。P3 は P2 のエイジングを部品として使える。
2. **P1 は最大価値だが要員アカウントという新概念を伴う**ため、P2/P3 で運用を安定させた後に
   専用 spec（F基盤→入力UI→承認→PDF のレーン分割）で着手。
3. **P4・P5 は営業サイドの即効薬**として P1 と並行可能（担当ファイルが交差しない）。
4. **P6・P7 は影響範囲が広い**ため、要望が具体化してから。P6 は `resolveContractAmount` への
   一点改修が成立するかの技術検証（スパイク）を先に1タスクだけ切るとよい。

# 実装全体計画（7 spec の実行順・競合・採番の正）

各 spec の tasks.md はレーン内の順序を定める。**spec 間の順序・競合・マイグレーション採番は
本章が唯一の正**とする（各 design の記述と食い違ったら本章を優先し、気づいた者が両方を直す）。

## Wave 構成

```
Wave 1（並行可）: P2 ar-management ∥ P4 quotation-management ∥ P5 revenue-forecast
Wave 2（並行可）: P3 monthly-closing-checklist ∥ P1 engineer-self-service-timesheet
Wave 3        : P6 contract-price-history
Wave 4（単独）  : P7 data-scope-permission
```

- Wave 1 の3件は担当ファイルが非交差（P2=請求系 / P4=見積新設+`ContractServiceImpl` /
  P5=ダッシュボード系）。※P4 レーンFの `ContractServiceImpl` は Wave 1 内で唯一の共有懸念だが
  P2/P5 は触らないため安全。
- P3 は P2 レーンA マージ後（`InvoiceMapper` 交差・残高定義の共用）。
- P1 は Wave 1 と非交差だが規模が大きいため Wave 2 に置く（Wave 1 と同時に始めてもよい。
  その場合も P6 より先に完了させること）。
- P6 は P1 完了後（`WorkRecordServiceImpl.saveHours` 交差）。
- P7 は全完了後・**他 spec と並行禁止**（全読み経路に触れるため）。

## マイグレーション採番の予約

**番号は Wave 順（=想定マージ順）に割り当てる**（Flyway は既定で out-of-order を拒否するため、
後の Wave に若い番号を与えてはならない）:

| 番号 | Wave | spec | 内容 |
|---|---|---|---|
| V28 | 1 | ar-management | t_invoice_payment・status ENUM・(必要なら顧客メール) |
| V29 | 1 | quotation-management | t_quotation・t_contract.quotation_id・メニューシード |
| V30 | 1 | revenue-forecast | config シードのみ |
| V31 | 2 | monthly-closing-checklist | メニューシードのみ |
| V32 | 2 | engineer-self-service-timesheet | ロール/勤怠 ENUM・2テーブル・メニューシード |
| V33 | 3 | contract-price-history | t_contract_price_history |
| V34 | 4 | data-scope-permission | config シードのみ |

**運用規則**:
1. 同一 Wave 内の並行 spec（V28〜V30）は**マイグレーションを含むレーン（各 tasks の最初の
   実装タスク）だけ番号順にマージ**し、以降のレーンは自由に並行する（乱序適用の回避）。
2. マージ順が予約と入れ替わる場合は本表を先に更新し、各 spec の design 内の番号表記も
   気づいた者が追随修正する（番号の重複・欠番の黙認は禁止）。
3. 実装時に既存最新が V27 でなくなっていた場合（他specの割り込み）は、本表全体を
   その時点の最新+1 から振り直す。

## 横断の共通規則（全 spec の tasks に共通で適用）

1. 新テーブル/カラムは H2 同期3点セット（engineer-schema-h2 / `sql/*-h2.sql` / application-test.yml）
   と `FlywayMigrationSmokeTest` の assert を**同一コミット**に含める。
2. 通知リンクは必ず `NotificationLinks` 定数へ（`NotificationLinkRouteTest` が自動検証）。
3. セレクト用途の一覧 API 呼び出しは `size=1000` を明示（money-flow F2 の教訓）。
4. 状態機械はサーバ側 Map が唯一の正・フロントはミラー定数＋相互参照コメント（contract の前例）。
5. 各 spec 完了時に `mvn test` 全緑を確認し、`.kiro/specs/README.md` の状態列と本章の
   Wave 進捗を更新する。Docker あり環境での smoke test は各 Wave 完了時にまとめて実行でよい。
