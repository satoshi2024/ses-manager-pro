# Requirements — 自身のパスワード変更(self-service-password)

現状、パスワードを変更できるのは管理者(ユーザー管理画面で他人の分も含めて)のみで、
`営業`/`HR`/`マネージャー` ロールの利用者は自分のパスワードを変更する手段が一切ない。
全ロールが使えるプロフィール機能(パスワード変更)を追加する。

対象コード: `controller/api/ProfileApiController.java`(新規) / `layout/header.html` /
`layout/base.html` / `static/js/profile.js`(新規)。

## R1. パスワード変更API

受け入れ基準:
1. `PUT /api/profile/password` が JSON `{currentPassword, newPassword}` を受け付ける。
   認証済みであればロールを問わず利用できる(`m_menu` への登録は不要 —
   `MenuPermissionFilter` は `m_menu` に一致しないパスを許可する。`/api/files` と同じ扱い)。
2. 現在のパスワードが一致しない場合は `BusinessException`「現在のパスワードが正しくありません」。
   照合は `PasswordEncoder.matches()` で行う(プロファイルにより平文/BCrypt が切り替わる
   既存構成のまま動くこと)。
3. 新パスワードは既存ポリシー(8文字以上・英字と数字を含む)を満たさない場合拒否される。
   ポリシー検証ロジックは `UserApiController.validatePasswordPolicy` と**重複実装しない**
   (共通化する)。
4. 新パスワードが現在のパスワードと同一の場合は「現在と同じパスワードは設定できません」で拒否。
5. 成功時、パスワードは `PasswordEncoder.encode()` で保存され、**セッションは維持される**
   (再ログイン不要)。
6. 対象ユーザーは常に「ログイン中の本人」(`Authentication` から解決)。リクエストボディで
   他ユーザーを指定する手段を持たない。

## R2. ヘッダーからの変更UI

受け入れ基準:
1. ヘッダー右上のユーザーメニュー(ドロップダウン)に「パスワード変更」項目が追加される。
   全ロール・全ページで表示される。
2. クリックで Bootstrap モーダル(現在のパスワード / 新しいパスワード / 新しいパスワード(確認)
   の3フィールド)が開く。
3. 新パスワードと確認の不一致はクライアント側で即時エラー表示(サーバーへ送らない)。
4. 成功時: モーダルが閉じ、成功 Toast「パスワードを変更しました」。
5. 失敗時: `ApiResult.message` を Toast 表示し、モーダルは開いたまま。
6. モーダルとJSはレイアウト層(`layout/base.html` / `header.html` + `js/profile.js`)に置き、
   全ページで動作する。

## R3. 監査

受け入れ基準:
1. パスワード変更の成功が既存の `ApiAuditFilter` の記録対象(`/api/**` の更新系)として
   監査ログに残ることを確認する(追加実装は不要のはずだが、テストで担保する)。
2. 監査ログ・アプリログのいずれにもパスワード平文が出力されない。
