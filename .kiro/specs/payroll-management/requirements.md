# Requirements Document — 給与計算(ヒアリング事項・未着手)

## ステータス: 未確定 — design.md / tasks.md は未作成

給与計算は日本の労働基準法・社会保険料計算・年末調整・自社雇用/BP契約の別など、法令対応と業務要件の確認が不可欠な領域であり、コードベース調査だけで仕様を確定できない。本ドキュメントは実装着手前に事業側へ確認すべき事項のチェックリストとして起票する。**design.md/tasks.mdは、以下の事項が確定してから作成する。**

## ヒアリング事項

### 1. スコープの確認
- [ ] 対象は「自社雇用エンジニアの給与計算」のみか、「BP(協力会社)への支払管理」（既存`t_bp_payment`で一部カバー済み）も含むか？
- [ ] 給与計算は自社内製で行うか、既存の外部給与システム(例: freee人事労務、SmartHR、弥生給与等)と連携する前提か？ 内製の場合、社会保険料率等のマスタデータ更新運用まで本システムで担うのは現実的なコストか要検討。

### 2. 法令対応範囲
- [ ] 所得税の源泉徴収計算・住民税の特別徴収対応は必要か？
- [ ] 社会保険料(健康保険・厚生年金・雇用保険)の計算は必要か？ 標準報酬月額の改定(定時決定・随時改定)への対応は？
- [ ] 年末調整機能は必要か？(扶養控除申告書の管理等、税務要件が複雑)

### 3. 既存データとの関係
- [ ] `t_engineer`は現状「稼働管理対象の技術者」であり雇用形態(正社員/契約社員/BP)を明示するカラムがない。給与計算対象を`t_engineer`から絞り込む場合、雇用形態カラムの追加が前提になる。
- [ ] `t_work_record`の`actualHours`(精算用実働時間)をそのまま給与計算の勤怠時間として流用してよいか、それとも別途正式な勤怠管理(休憩時間・深夜/休日割増等)が必要か？

### 4. セキュリティ・権限
- [ ] 給与情報は既存のどのロール(管理者のみ？HR？)がアクセスできるべきか？ 既存の`ROLE_管理者/営業/HR/マネージャー`の枠組みでは営業・マネージャーに給与情報が見えてはならない可能性が高く、新規の権限区分が必要になる可能性がある。

## 踩坑点（今後の設計時の注意）
- 給与計算を自社内製する場合、税制・保険料率は毎年改定されるため、**マスタデータの年次メンテナンス運用**を誰が担うかを決めずに実装すると、リリース後すぐに計算が古い料率のまま止まるリスクが高い。外部給与システム連携（CSV出力連携等）で済ませる選択肢を先に検討することを推奨する。
- 給与明細には氏名・口座番号等の高度な個人情報が含まれるため、既存の`AuditLog`/権限モデルだけで十分か、暗号化保存等の追加対策が必要かを情報セキュリティ担当と確認すること。
